# Preamble

Successful software projects are built through a disciplined and repeatable engineering process.

The Software Development Lifecycle (SDLC) Framework defines the standard lifecycle adopted across all TechAndMe projects, ensuring that every project progresses through clearly defined stages while maintaining engineering quality, governance, and documentation.

The framework integrates planning, architecture, implementation, testing, deployment, and continuous improvement into a unified engineering workflow.

---

# 1. Purpose

The purpose of this framework is to establish a common software development lifecycle for all TechAndMe projects.

The framework aims to:

- Improve engineering consistency.
- Reduce project risk.
- Encourage architectural thinking.
- Promote documentation-driven development.
- Ensure quality throughout the project lifecycle.
- Support continuous improvement.

---

# 2. Scope

This framework applies to:

- Software applications.
- AI-enabled systems.
- Internal engineering tools.
- Open-source projects.
- Research and prototype projects where appropriate.

It provides the overall lifecycle while allowing individual projects to tailor implementation details to their specific needs.

---

# 3. SDLC Philosophy

The TechAndMe Software Development Lifecycle is founded on several principles.

- Engineering before programming.
- Architecture before implementation.
- Documentation before complexity.
- Quality throughout development.
- Continuous review and improvement.
- Knowledge preserved through documentation.
- Checkpoints that support long-term continuity.

Every lifecycle stage contributes to the overall quality of the finished product.

---

# 4. Lifecycle Stages

Every TechAndMe project should progress through the following lifecycle.

## Stage 1 — Vision and Discovery

Define:

- Business problem.
- Stakeholders.
- Objectives.
- Scope.
- Success criteria.

---

## Stage 2 — Planning

Develop:

- Project roadmap.
- Business documentation.
- Initial requirements.
- Risk assessment.

---

## Stage 3 — Architecture

Produce:

- Architecture documentation.
- System design.
- Technology selection.
- Architecture reviews.
- Decision records.

Implementation should not begin until architecture has been reviewed.

---

## Stage 4 — Implementation

Develop the software according to the approved architecture.

Implementation should emphasise:

- Clean code.
- Modular design.
- Engineering standards.
- Version control.
- Continuous documentation.

---

## Stage 5 — Testing

Verify that the system satisfies its intended requirements.

Testing activities may include:

- Unit testing.
- Integration testing.
- System testing.
- User acceptance testing.
- Performance testing.
- Regression testing.

---

## Stage 6 — Deployment

Prepare the software for operational use.

Typical activities include:

- Packaging.
- Release preparation.
- Deployment.
- Environment validation.
- Release documentation.

---

## Stage 7 — Maintenance

Continue improving the system through:

- Bug fixes.
- Performance improvements.
- Security updates.
- Documentation updates.
- Feature enhancements.

Maintenance is considered part of the engineering lifecycle rather than a separate activity.

---

# 5. Stage Deliverables

Each lifecycle stage should produce identifiable outputs.

| Stage | Typical Deliverables |
|---------|----------------------|
| Vision | Vision, objectives, stakeholder analysis |
| Planning | Requirements, roadmap, risks |
| Architecture | Architecture documents, ADRs, reviews |
| Implementation | Source code, documentation, checkpoints |
| Testing | Test plans, results, defect reports |
| Deployment | Release package, deployment guide |
| Maintenance | Updates, improvements, release notes |

Deliverables provide measurable evidence of project progress.

---

# 6. Engineering Gates

Progress between lifecycle stages should occur only after defined engineering gates have been satisfied.

Typical gates include:

- Vision approved.
- Requirements reviewed.
- Architecture accepted.
- Implementation stable.
- Testing completed.
- Deployment approved.

Engineering gates improve quality by ensuring that each stage is completed before the next begins.

---

# 7. Continuous Improvement

Continuous improvement occurs throughout the lifecycle.

Projects should regularly:

- Review architecture.
- Refine documentation.
- Improve engineering standards.
- Capture lessons learned.
- Update reusable patterns.
- Record engineering decisions.

Improvement is expected throughout the lifecycle rather than after project completion.

Consistent engineering processes create consistent engineering outcomes.
